# AGENTS.md

## Project Overview

This is an **unofficial** Go SDK for the
[PayPay Open Payment API (OPA)](https://developer.paypay.ne.jp).
The package name is `paypayopa` and the module path is
`github.com/mythrnr/paypayopa-sdk-go`.

The SDK is organized around **integration scenarios**
(Web Payment, Native Payment, Dynamic QR, etc.)
rather than raw REST API endpoints.
Each scenario has a dedicated struct
that wraps an internal HTTP client.

## Requirements

- Go 1.24 or above
- Docker (for linting, mock generation, spell checking, and markdown linting)

## Repository Structure

```text
*.go                    # All source and test files live at the package root
internal/
  signer.go             # HMAC-SHA256 request signing
  nonce.go              # Random nonce generation
  mocks/                # Generated mocks (do NOT edit by hand)
.github/workflows/      # CI pipelines
.golangci.yaml          # Linter configuration (golangci-lint v2 format)
.mockery.yaml           # Mock generation configuration (mockery v3)
.editorconfig           # Editor formatting rules
Makefile                # Build, test, lint, and release targets
```

### Key Source Files

| File | Purpose |
| ---- | ------- |
| `client.go` | Internal HTTP client (`opaClient`) and auth interceptor |
| `credentials.go` | `Credentials` struct and `NewCredentials` factory |
| `constants.go` | Type definitions, enums, and constants |
| `jwt.go` | JWT token parsing and validation |
| `webhook.go` | Webhook payload data structures |
| `api_*.go` | Internal API call implementations (payment, qrcode, refund, user, wallet) |
| `scenario_*.go` | Public integration clients (WebPayment, NativePayment, DynamicQR, etc.) |

## Architecture

### Client Pattern

1. **`opaClient`** (internal) wraps `*http.Client`
   and provides `GET`, `POST`, `DELETE` methods.
   All methods accept `context.Context`
   and return `(*ResultInfo, error)`.
2. **`authInterceptor`** implements `http.RoundTripper`
   and signs every request with HMAC-SHA256
   using the credentials provided at construction time.
3. **Scenario structs** (e.g., `WebPayment`, `DynamicQR`) are the public API surface.
   Each is constructed via `NewXxx(creds)` or `NewXxxWithHTTPClient(creds, httpClient)`.

### Response Convention

All public API methods return `(data, *ResultInfo, error)`.
Use `info.Success()` (HTTP status < 400) to check for API-level errors.

### Error Handling

- Use `fmt.Errorf` with `%w` for wrapping.
- `panic()` is used only for programmer errors (nil credentials or nil client).
- `ErrJWTAudienceNotMatch` is the only exported sentinel error.

## Code Conventions

### Language

- Code comments: bilingual (English and Japanese) for public symbols.
- Log messages: English.
- Error messages: English.

### Naming

- Standard Go naming conventions apply.
- Test files use `_inner_test.go` suffix (not `_test.go`) to stay in the same package
  while avoiding the `testpackage` linter. This is configured in `.golangci.yaml`.

### Formatting

- Line length limit: 100 characters.
- Indent with tabs for `.go` files, spaces for others (see `.editorconfig`).
- Formatters: `gci`, `gofmt`, `gofumpt`, `goimports` (enforced by golangci-lint).

### Linting

- golangci-lint v2 with `default: all` minus a few disabled linters.
- Cyclomatic/cognitive complexity threshold: 20.
- Duplication threshold: 100 lines.
- `dupl`, `err113`, `funlen`, `maintidx` are excluded for test files.
- Security linter (`gosec`) excludes: G401, G404, G501
  (MD5/weak-random used intentionally for HMAC signing).

## Testing

### Patterns

- All tests are in `*_inner_test.go` files at the package root.
- Use `t.Parallel()` at both the top-level and sub-test level.
- Table-driven tests with `t.Run()`.
- `github.com/stretchr/testify` for assertions (`assert`, `require`) and mocks.

### Mocks

- Generated by [mockery](https://github.com/vektra/mockery) v3 (configured in `.mockery.yaml`).
- Three mocked interfaces: `json.Marshaler`, `io.ReadCloser`, `net/http.RoundTripper`.
- Output directory: `internal/mocks/`.
- **Never edit generated mock files by hand.** Regenerate with `make mock`.

### Running Tests

```bash
make test              # Run all tests with coverage
make test-json         # Run tests with JSON output (used in CI)
```

## Development Workflow

### Makefile Targets

| Target | Description |
| ------ | ----------- |
| `make ci-suite` | Full CI pipeline (spell-check, fmt, lint, lint-markdown, mock, vulnerability-check, test) |
| `make fmt` | Format code with `go fmt` |
| `make lint` | Run golangci-lint via Docker |
| `make lint-markdown` | Lint markdown files via Docker |
| `make mock` | Regenerate mocks via Docker |
| `make spell-check` | Run cspell spell checker via Docker |
| `make test` | Run tests with coverage |
| `make tidy` | Run `go mod tidy` |
| `make vulnerability-check` | Run `govulncheck` |
| `make release version=vX.Y.Z` | Create a GitHub release targeting `master` |

### CI Pipeline

- **golangci-lint**: Code style and static analysis.
- **spell-check**: Spelling validation with cspell.
- **testing**: Runs on Go 1.24 and 1.25.
- **vulnerability scanning**: `govulncheck`, runs daily and on every push.

## Dependencies

Production dependencies are intentionally minimal:

- `github.com/golang-jwt/jwt/v5` - JWT token handling
- `github.com/stretchr/testify` - Testing only

The SDK relies on the Go standard library for HTTP, JSON, and cryptography.

## Release Process

Releases are created from the `master` branch using `make release version=vX.Y.Z`.
This runs `gh release create` with auto-generated notes.
